# kllmgate 详细设计

本文档基于[架构设计](架构设计.md)中确定的目标和边界，给出面向实现的详细设计。所有接口定义、字段映射和流程描述均以 V1 落地为准。

**关键设计决策：**

1. Converter 全权负责流式转换——每个 converter 自行解析上游 SSE、产出客户端 SSE，内部通过 helper 方法在流式与非流式之间共享转换逻辑
2. Provider 配置增加 `tool_style` 字段，用于选择工具调用适配器
3. 客户端协议与上游协议一致时，使用 PassthroughConverter 走正常管线，不做 pipeline 短路
4. V1 一次实现全部 6 条 pairwise 转换链路 + 直通

## 2. 核心数据模型

### 2.1 ProtocolFormat

标识一种完整的协议格式（协议 + wire API）：

```python
class ProtocolFormat(str, Enum):
    OPENAI_CHAT = "openai.chat"
    OPENAI_RESPONSES = "openai.responses"
    ANTHROPIC_MESSAGES = "anthropic.messages"
```

用途：入站识别、上游识别、converter 选择键。

### 2.2 ProviderConfig

```python
@dataclass
class ProviderConfig:
    name: str                       # provider 标识，对应 TOML 中的 key
    base_url: str                   # 上游 base URL，末尾不含 /
    api_key: str | None             # 直接配置的 API key，优先级高于 env_key
    env_key: str | None             # API key 的环境变量名
    protocol: str                   # "openai" | "anthropic"
    wire_api: str                   # "chat" | "responses" | "messages"
    tool_style: str                 # "standard" | "minimax_xml"
    timeout_seconds: int            # 请求超时（秒）
    max_retries: int                # 最大重试次数
    models: list[str] | None       # 模型白名单，None 表示不限制

    @property
    def protocol_format(self) -> ProtocolFormat:
        if self.protocol == "anthropic":
            return ProtocolFormat.ANTHROPIC_MESSAGES
        if self.wire_api == "responses":
            return ProtocolFormat.OPENAI_RESPONSES
        return ProtocolFormat.OPENAI_CHAT

    def resolve_api_key(self) -> str:
        """解析 provider 的 API key

        优先使用显式配置的 api_key；未配置时再读取 env_key 对应的环境变量。
        若两者都不可用，则抛出 ConfigError。
        """
```

**字段默认值与约束：**

| 字段 | 默认值 | 约束 |
|---|---|---|
| `api_key` | `None` | 存在时直接使用，优先级高于 `env_key` |
| `env_key` | `None` | `api_key` 未配置时，从此环境变量读取 |
| `wire_api` | `protocol="openai"` 时默认 `"chat"`；`protocol="anthropic"` 时固定 `"messages"` | `"chat"` / `"responses"` / `"messages"` |
| `tool_style` | `"standard"` | `"standard"` / `"minimax_xml"` |
| `timeout_seconds` | `120` | 正整数 |
| `max_retries` | `2` | 非负整数 |
| `models` | `None` | 字符串列表或不配置 |

## 3. 配置设计

### 3.1 配置文件格式

使用 TOML，默认路径 `config.toml`（可通过 `--config` 指定）：

```toml
[providers.openai_official]
base_url = "https://api.openai.com/v1"
env_key = "OPENAI_API_KEY"
protocol = "openai"
wire_api = "chat"
timeout_seconds = 120
max_retries = 2
models = ["gpt-4.1", "gpt-4o-mini"]

[providers.anthropic_official]
base_url = "https://api.anthropic.com"
env_key = "ANTHROPIC_API_KEY"
protocol = "anthropic"
timeout_seconds = 120
max_retries = 2
models = ["claude-sonnet-4-20250514"]

[providers.minimax_proxy]
base_url = "https://api.scnet.cn/api/llm/v1"
api_key = "sk-xxx"           # 直接配置 API key（优先级高于 env_key）
env_key = "SCNET_API_KEY"    # api_key 未配置时从此环境变量读取
protocol = "openai"
wire_api = "chat"
tool_style = "minimax_xml"
timeout_seconds = 120
max_retries = 5
```

### 3.2 加载与校验流程

启动时执行以下步骤，任一步骤失败则终止进程并输出明确错误信息：

1. 读取 TOML 文件（使用标准库 `tomllib`，要求 Python ≥ 3.11）
2. 遍历 `providers` 下的每个 key，构造 `ProviderConfig`
3. 校验 `protocol` 值合法
4. 校验 `wire_api` 与 `protocol` 的组合合法（`anthropic` 不允许配置 `wire_api` 为 `chat`/`responses`）
5. 解析 API key：若配置了 `api_key` 字段则直接使用；否则读取 `env_key` 指向的环境变量。两者都未配置或解析结果为空则报错
6. 去除 `base_url` 末尾的 `/`

校验通过后，为每个 provider 创建 `UpstreamClient` 实例，存入 `app.state`。

## 4. 请求处理管线

### 4.1 总流程

```python
async def process_request(
    inbound_format: ProtocolFormat,
    body: dict,
) -> JSONResponse | StreamingResponse:
    # 1. 提取并校验 model 字段
    model_ref = body.get("model", "")
    if not model_ref or "/" not in model_ref:
        raise ConfigError(f"model 必须为 provider/model 格式: {model_ref!r}")

    # 2. 解析 provider 与上游模型名
    provider_name, upstream_model = model_ref.split("/", 1)
    provider = get_provider(provider_name)  # 不存在则抛 ConfigError
    validate_model(provider, upstream_model)  # 白名单校验

    # 3. 选择 converter 和 tool adapter
    upstream_format = provider.protocol_format
    tool_adapter = get_tool_adapter(provider)
    converter = get_converter(inbound_format, upstream_format, tool_adapter)

    # 4. 转换请求
    upstream_body = converter.convert_request(body, upstream_model)

    # 5. 获取上游客户端并发送
    client = get_upstream_client(provider.name)
    stream = body.get("stream", False)

    if stream:
        upstream_events = client.send_stream(upstream_body)
        event_stream = converter.convert_stream(upstream_events)
        return StreamingResponse(
            event_stream,
            media_type="text/event-stream",
            headers={"Cache-Control": "no-cache", "X-Accel-Buffering": "no"},
        )

    upstream_response = await client.send(upstream_body)
    return JSONResponse(converter.convert_response(upstream_response))
```

### 4.2 错误拦截

在入口路由层捕获 `GatewayError`，根据 `inbound_format` 格式化为对应协议的错误响应（详见第 10 节）。

### 4.3 流式错误处理

流式请求始终返回 HTTP 200。错误通过 SSE 事件传达：

- 上游不可达/超时：converter 在 `convert_stream` 中捕获异常，产出包含错误信息的完成事件
- 上游返回 HTTP 错误：upstream client 在 `send_stream` 中检测状态码，抛出异常由 converter 捕获
- 一旦开始输出客户端 SSE 事件，不再切换 HTTP 状态码，后续错误统一落入协议内的流式错误事件

## 5. 入口路由层

三条路由，对应三种客户端协议。每条路由负责捕获 `GatewayError` 并按自身协议格式化错误响应（`inbound_format` 在路由层已知，无需额外传递）：

```python
@app.post("/openai/chat/completions")
async def openai_chat(request: Request):
    try:
        body = await request.json()
        return await process_request(ProtocolFormat.OPENAI_CHAT, body)
    except GatewayError as e:
        return format_error_response(e, ProtocolFormat.OPENAI_CHAT)

@app.post("/openai/responses")
async def openai_responses(request: Request):
    try:
        body = await request.json()
        return await process_request(ProtocolFormat.OPENAI_RESPONSES, body)
    except GatewayError as e:
        return format_error_response(e, ProtocolFormat.OPENAI_RESPONSES)

@app.post("/anthropic/v1/messages")
async def anthropic_messages(request: Request):
    try:
        body = await request.json()
        return await process_request(ProtocolFormat.ANTHROPIC_MESSAGES, body)
    except GatewayError as e:
        return format_error_response(e, ProtocolFormat.ANTHROPIC_MESSAGES)
```

路由层不做业务逻辑，仅负责提取请求体、调用管线和格式化错误响应。

## 6. 模型解析与路由

### 6.1 解析规则

按首个 `/` 分割 `model` 字段：

```text
openai_official/gpt-4.1
^^^^^^^^^^^^^^^^ ^^^^^^^
provider_name    upstream_model
```

### 6.2 校验流程

1. `provider_name` 不在已加载配置中 → `ConfigError("unknown provider")`
2. provider 配置了 `models` 白名单且 `upstream_model` 不在列表中 → `ConfigError("model not supported")`
3. `api_key` 和 `env_key` 均未配置或解析为空 → `ConfigError("API key not configured")`

### 6.3 Converter 选择

Converter 注册表是一个以 `(inbound_format, upstream_format)` 为键的字典：

```python
CONVERTER_REGISTRY: dict[
    tuple[ProtocolFormat, ProtocolFormat],
    type[Converter],
] = {
    (PF.OPENAI_CHAT, PF.OPENAI_RESPONSES): OpenaiChatToOpenaiResponsesConverter,
    (PF.OPENAI_CHAT, PF.ANTHROPIC_MESSAGES): OpenaiChatToAnthropicMessagesConverter,
    (PF.OPENAI_RESPONSES, PF.OPENAI_CHAT): OpenaiResponsesToOpenaiChatConverter,
    (PF.OPENAI_RESPONSES, PF.ANTHROPIC_MESSAGES): OpenaiResponsesToAnthropicMessagesConverter,
    (PF.ANTHROPIC_MESSAGES, PF.OPENAI_CHAT): AnthropicMessagesToOpenaiChatConverter,
    (PF.ANTHROPIC_MESSAGES, PF.OPENAI_RESPONSES): AnthropicMessagesToOpenaiResponsesConverter,
}
```

当以下条件同时满足时使用 `PassthroughConverter`，不查表：

1. `inbound_format == upstream_format`
2. `provider.tool_style == "standard"`（工具调用协议无需适配）

若协议相同但 `tool_style != "standard"`（如 MiniMax），仍需走注册表中 `(inbound_format, inbound_format)` 对应的同协议转换器。该转换器消息结构不做变换，只通过 tool_adapter 处理工具定义注入和工具调用解析。因此注册表需补充同协议条目：

```python
(PF.OPENAI_CHAT, PF.OPENAI_CHAT): OpenaiChatToolAdaptConverter,
(PF.OPENAI_RESPONSES, PF.OPENAI_RESPONSES): OpenaiResponsesToolAdaptConverter,
```

Anthropic 不存在 tool_style 变体，`(PF.ANTHROPIC_MESSAGES, PF.ANTHROPIC_MESSAGES)` 始终走 PassthroughConverter。

每次请求创建新的 converter 实例（因为流式转换需要维护请求级状态）。

### 6.4 Tool Adapter 选择

```python
def get_tool_adapter(provider: ProviderConfig) -> ToolAdapter:
    if provider.protocol == "anthropic":
        return AnthropicToolAdapter()

    match provider.tool_style:
        case "minimax_xml":
            return MinimaxXmlToolAdapter()
        case "standard":
            return StandardToolAdapter()
        case _:
            raise ConfigError(
                f"unknown tool_style for provider {provider.name!r}: "
                f"{provider.tool_style!r}"
            )
```

当上游为 Anthropic 协议时，tool adapter 固定使用 `AnthropicToolAdapter`，忽略 `tool_style` 配置（Anthropic 有自己的工具格式，不存在厂商变体）。

## 7. Converter 层

### 7.1 接口定义

```python
class Converter(ABC):
    def __init__(self, tool_adapter: ToolAdapter):
        self.tool_adapter = tool_adapter

    @abstractmethod
    def convert_request(self, body: dict, model: str) -> dict:
        """将客户端请求体转换为上游请求体

        Args:
            body: 客户端发送的原始请求体
            model: 去掉 provider 前缀后的上游模型名

        Returns:
            可直接发送给上游的请求体
        """

    @abstractmethod
    def convert_response(self, response: dict) -> dict:
        """将上游非流式响应转换为客户端协议格式"""

    @abstractmethod
    async def convert_stream(
        self,
        upstream_events: AsyncIterator[SseEvent],
    ) -> AsyncIterator[str]:
        """将上游 SSE 流转换为客户端协议的 SSE 流

        接收上游 HTTP 响应按帧解析后的 SSE 事件迭代器，产出客户端可直接
        消费的 SSE 文本块。
        converter 负责协议级事件映射与状态维护；底层分帧由 sse.py 和
        upstream client 负责。
        """
```

**Converter 与 ToolAdapter 的职责边界：**

- Converter 负责协议结构的映射（消息格式、字段名、事件类型等）
- ToolAdapter 负责工具相关字段的格式转换（工具定义、工具调用、工具结果）
- Converter 在遇到工具相关字段时调用 ToolAdapter 的对应方法

### 7.2 PassthroughConverter

当 `inbound_format == upstream_format` 时使用。职责：

- `convert_request`：仅替换 `model` 字段为去除 provider 前缀后的值，其他字段原样保留
- `convert_response`：原样返回
- `convert_stream`：将每个 `SseEvent` 重新编码为 SSE 文本块后原样转发

即使是直通，也走正常管线，确保日志记录、错误处理路径一致。

### 7.3 各转换链路

以下为每条链路的核心字段映射。converter 同时负责请求方向和响应方向的转换。

#### 7.3.1 openai_responses_to_openai_chat

当前代码中最成熟的链路，重构到新架构即可。

**请求转换（Responses → Chat）：**

| Responses API | Chat Completions | 说明 |
|---|---|---|
| `model` | `model` | 去除 provider 前缀 |
| `instructions` | 合并到 `messages[0]`（role=system） | |
| `input` 中 role 为 user/assistant 的消息 | `messages` 中对应 role 消息 | role `"model"` → `"assistant"`，`"developer"` → 合并到 system |
| `input` 中 `type: "function_call"` | 委托 tool_adapter.make_tool_calls_message | |
| `input` 中 `type: "function_call_output"` | 委托 tool_adapter.make_tool_result_message | |
| `tools` | 委托 tool_adapter.convert_tool_definitions | |
| `stream` | `stream` | 若为 true，添加 `stream_options.include_usage = true` |
| content 为 list 的消息 | 转为纯文本或结构化 content | 多模态内容按协议做结构映射 |

**响应转换（Chat → Responses）：**

| Chat Completions | Responses API | 说明 |
|---|---|---|
| `id` | `id` | |
| `created` | `created_at` | |
| `model` | `model` | |
| `choices[0].message.content` | `output[].type="message"` → `content[].type="output_text"` | 经 tool_adapter.extract_tool_calls 清理后的文本 |
| tool_adapter 提取的工具调用 | `output[].type="function_call"` | 包含 id、call_id、name、arguments |
| `choices[0].finish_reason` | `status` | 映射见下表 |
| `usage.prompt_tokens` | `usage.input_tokens` | |
| `usage.completion_tokens` | `usage.output_tokens` | |

`finish_reason` → `status` 映射：

| Chat `finish_reason` | Responses `status` |
|---|---|
| `stop` | `completed` |
| `tool_calls` | `completed` |
| `length` | `incomplete` |
| `content_filter` | `incomplete` |

**流式转换（Chat SSE → Responses SSE）：**

上游事件流仅有 `data:` 行，需要映射为 Responses API 的多事件格式：

| 时机 | 产出的 Responses 事件 |
|---|---|
| 流开始 | `response.created` → `response.in_progress` → `response.output_item.added` → `response.content_part.added` |
| 收到 `delta.content` | `response.output_text.delta`（需配合 tool_adapter.detect_stream_tool_boundary 做缓冲） |
| 流结束后的文本 | `response.output_text.done` → `response.content_part.done` → `response.output_item.done` |
| 流结束后的工具调用 | 每个调用产出 `response.output_item.added` → `response.function_call_arguments.delta` → `response.function_call_arguments.done` → `response.output_item.done` |
| 最终 | `response.completed`（包含完整 response 对象和 usage） |

每个事件的 `data` 中需要递增 `sequence_number`。

#### 7.3.2 openai_chat_to_openai_responses

7.3.1 的反向链路。

**请求转换（Chat → Responses）：**

| Chat Completions | Responses API 上游请求 | 说明 |
|---|---|---|
| `messages` 中 role=system | `instructions` | 收集全部 system 消息并按原顺序拼接 |
| `messages` 中非 system 消息 | `input` 中的消息 item | role `"assistant"` → `"model"` |
| `messages` 中 role=assistant 且含 tool_calls | `input` 中 `type: "function_call"` items | |
| `messages` 中 role=tool | `input` 中 `type: "function_call_output"` items | |
| `tools` | `tools` | OpenAI 格式不变，直接传递 |
| `stream` | `stream` | |

**响应转换（Responses → Chat）：**

| Responses API | Chat Completions | 说明 |
|---|---|---|
| `output[].type="message"` → `content[].text` | `choices[0].message.content` | |
| `output[].type="function_call"` | `choices[0].message.tool_calls[]` | |
| `status` | `choices[0].finish_reason` | 映射见下表 |
| `usage.input_tokens` | `usage.prompt_tokens` | |
| `usage.output_tokens` | `usage.completion_tokens` | |

`status` → `finish_reason` 映射：

| Responses `status` | Chat `finish_reason` |
|---|---|
| `completed`（输出含 function_call） | `tool_calls` |
| `completed`（输出不含 function_call） | `stop` |
| `incomplete` | `length` |
| `failed` | `stop` |

**流式转换（Responses SSE → Chat SSE）：**

| 上游 Responses 事件 | 产出的 Chat 事件 |
|---|---|
| `response.output_text.delta` | `data: {"choices": [{"delta": {"content": delta_text}}]}` |
| `response.function_call_arguments.delta` | `data: {"choices": [{"delta": {"tool_calls": [...]}}]}` |
| `response.completed` | 发送 usage chunk + `data: [DONE]` |

#### 7.3.3 openai_chat_to_anthropic_messages

**请求转换（OpenAI Chat → Anthropic Messages）：**

| OpenAI Chat | Anthropic Messages | 说明 |
|---|---|---|
| `messages` 中 role=system 的 content | `system`（顶层字段） | Anthropic 的 system 不在 messages 中 |
| `messages` 中 role=user/assistant | `messages` | |
| `messages` 中 role=assistant 且含 `tool_calls` | assistant 消息中 `tool_use` content block | 委托 tool_adapter |
| `messages` 中 role=tool | user 消息中 `tool_result` content block | 委托 tool_adapter |
| `tools` | `tools` | 委托 tool_adapter（`parameters` → `input_schema`） |
| `model` | `model` | |
| `stream` | `stream` | |
| `max_tokens` / `max_completion_tokens` | `max_tokens` | Anthropic 要求此字段，若客户端未传则使用默认值 4096 |
| `temperature` | `temperature` | 可选，直接传递 |
| `top_p` | `top_p` | 可选，直接传递 |

**响应转换（Anthropic Messages → OpenAI Chat）：**

| Anthropic Messages | OpenAI Chat | 说明 |
|---|---|---|
| `id` | `id` | |
| —— | `created` | 网关生成当前时间戳 |
| `model` | `model` | |
| `content[]` 中 type=text | `choices[0].message.content` | 拼接所有 text block |
| `content[]` 中 type=tool_use | `choices[0].message.tool_calls[]` | 委托 tool_adapter |
| `stop_reason` | `choices[0].finish_reason` | 映射见下表 |
| `usage.input_tokens` | `usage.prompt_tokens` | |
| `usage.output_tokens` | `usage.completion_tokens` | |

`stop_reason` 映射：

| Anthropic | OpenAI |
|---|---|
| `end_turn` | `stop` |
| `tool_use` | `tool_calls` |
| `max_tokens` | `length` |
| `stop_sequence` | `stop` |

**流式转换（Anthropic SSE → OpenAI Chat SSE）：**

| 上游 Anthropic 事件 | 产出的 Chat 事件 |
|---|---|
| `message_start` | `data: {"choices": [{"delta": {"role": "assistant"}}]}` |
| `content_block_delta`（type=text_delta） | `data: {"choices": [{"delta": {"content": text}}]}` |
| `content_block_delta`（type=input_json_delta） | 缓冲工具调用参数，在 `content_block_stop` 时发送完整 `tool_calls` delta |
| `message_delta` | `data: {"choices": [{"delta": {}, "finish_reason": mapped_reason}], "usage": {...}}` |
| `message_stop` | `data: [DONE]` |

#### 7.3.4 openai_responses_to_anthropic_messages

请求方向可视为两步组合：先将 Responses API input 解析为中间消息列表（复用 7.3.1 的 input 解析逻辑），再将消息列表转换为 Anthropic 格式（复用 7.3.3 的消息映射逻辑）。

响应方向将 Anthropic 响应转换为 Responses API 格式，可参考 7.3.3 的响应字段映射 + 7.3.1 的 Responses 输出构建。

流式方向：上游为 Anthropic SSE 事件，需映射为 Responses API 事件序列（参考 7.3.1 的 Responses 事件结构）。

#### 7.3.5 anthropic_messages_to_openai_chat

7.3.3 的反向链路。

**请求转换（Anthropic Messages → OpenAI Chat）：**

| Anthropic Messages | OpenAI Chat | 说明 |
|---|---|---|
| `system` | `messages[0]`（role=system） | |
| `messages` 中 role=user/assistant | `messages` | |
| assistant 消息中 `tool_use` content block | role=assistant + `tool_calls` 字段 | 委托 tool_adapter |
| user 消息中 `tool_result` content block | role=tool 消息 | 委托 tool_adapter |
| `tools` | `tools` | 委托 tool_adapter（`input_schema` → `parameters`） |
| `max_tokens` | `max_tokens` | |

**响应转换（OpenAI Chat → Anthropic Messages）：**

| OpenAI Chat | Anthropic Messages | 说明 |
|---|---|---|
| `choices[0].message.content` | `content[]` type=text | |
| `choices[0].message.tool_calls` | `content[]` type=tool_use | 委托 tool_adapter |
| `choices[0].finish_reason` | `stop_reason` | 反向映射 |
| `usage.prompt_tokens` | `usage.input_tokens` | |
| `usage.completion_tokens` | `usage.output_tokens` | |

**流式转换（OpenAI Chat SSE → Anthropic SSE）：**

| 上游 Chat 事件 | 产出的 Anthropic 事件 |
|---|---|
| 首个 delta | `message_start` + `content_block_start` |
| `delta.content` | `content_block_delta`（type=text_delta） |
| `delta.tool_calls` | `content_block_start`（type=tool_use）+ `content_block_delta`（type=input_json_delta） |
| `finish_reason` | `content_block_stop` + `message_delta`（含 stop_reason）+ `message_stop` |

#### 7.3.6 anthropic_messages_to_openai_responses

请求方向将 Anthropic 请求解析为中间消息列表（复用 7.3.5 的请求解析逻辑），再组装为 Responses API 上游请求。

响应方向将 Responses API 响应转换为 Anthropic Messages 格式（参考 7.3.5 的响应映射 + Responses 输出解析）。

流式方向：上游为 Responses SSE 事件，映射为 Anthropic SSE 事件序列。

## 8. 工具调用适配层

### 8.1 接口定义

ToolAdapter 封装上游特定的工具格式差异，供 converter 在处理工具相关字段时调用：

```python
class ToolAdapter(ABC):
    @abstractmethod
    def convert_tool_definitions(
        self, tools: list[dict],
    ) -> tuple[str | None, list[dict] | None]:
        """转换工具定义为上游格式

        Returns:
            (system_prompt_addition, tools_field)
            - 当工具定义走 API 字段时：返回 (None, converted_tools)
            - 当工具定义走 system prompt 时：返回 ("prompt text", None)
        """

    @abstractmethod
    def make_tool_calls_message(
        self, calls: list[dict],
    ) -> dict:
        """将历史工具调用格式化为上游可接受的 assistant 消息

        Args:
            calls: [{"name": str, "arguments": str, "call_id": str}, ...]

        Returns:
            一条 assistant 角色的消息 dict
        """

    @abstractmethod
    def make_tool_result_message(
        self, call_id: str, name: str, output: str,
    ) -> dict:
        """将工具执行结果格式化为上游可接受的消息"""

    @abstractmethod
    def extract_tool_calls(
        self, message: dict,
    ) -> tuple[str, list[dict]]:
        """从上游响应消息中提取工具调用

        Returns:
            (cleaned_content, normalized_calls)
            normalized_calls 格式: [{"id": str, "name": str, "arguments": str}, ...]
        """

    @abstractmethod
    def detect_stream_tool_boundary(
        self, text: str,
    ) -> int | None:
        """检测流式文本中是否出现工具调用边界

        Returns:
            工具调用起始标签的位置，或 None 表示无工具调用。
            用于流式转换时判断哪些文本可以安全发送给客户端。
        """
```

### 8.2 StandardToolAdapter

用于上游为标准 OpenAI Chat 或 Responses API 且 `tool_style = "standard"` 的场景。

| 方法 | 行为 |
|---|---|
| `convert_tool_definitions` | 返回 `(None, tools)`，原样传递 |
| `make_tool_calls_message` | 构造 `{"role": "assistant", "tool_calls": [{"id": call_id, "type": "function", "function": {"name": name, "arguments": arguments}}]}` |
| `make_tool_result_message` | 构造 `{"role": "tool", "tool_call_id": call_id, "content": output}` |
| `extract_tool_calls` | 从 `message["tool_calls"]` 提取，content 原样返回 |
| `detect_stream_tool_boundary` | 始终返回 `None`（标准格式的工具调用通过结构化 delta 传递，不嵌入文本） |

### 8.3 MinimaxXmlToolAdapter

用于上游为 MiniMax 系列模型且 `tool_style = "minimax_xml"` 的场景。核心逻辑从当前 `kllmgate.py` 中迁移。

| 方法 | 行为 |
|---|---|
| `convert_tool_definitions` | 将工具定义转为 XML prompt 文本，返回 `(prompt_text, None)` |
| `make_tool_calls_message` | 构造 `{"role": "assistant", "content": "<minimax:tool_call>..XML..</minimax:tool_call>"}` |
| `make_tool_result_message` | 构造 `{"role": "user", "content": "[Tool Result] {name}:\n{output}"}` |
| `extract_tool_calls` | 用正则从 `message["content"]` 中解析 `<minimax:tool_call>` XML，返回 (清理后文本, 解析出的调用列表) |
| `detect_stream_tool_boundary` | 查找 `<minimax:tool_call>` 标签位置，返回其起始索引 |

额外职责：

- **畸形工具调用检测**：XML 标签不闭合、函数名为空、参数非法 JSON 等情况，返回错误描述供 converter 标记 `status: "incomplete"`
- **残留片段清理**：当检测到畸形调用时，移除文本中残留的 XML 片段

### 8.4 AnthropicToolAdapter

用于上游为 Anthropic Messages API 的场景。当 `provider.protocol == "anthropic"` 时强制使用，不受 `tool_style` 影响。

| 方法 | 行为 |
|---|---|
| `convert_tool_definitions` | 将 OpenAI 格式转为 Anthropic 格式（`function.parameters` → `input_schema`，去除 `type: "function"` 包装），返回 `(None, converted_tools)` |
| `make_tool_calls_message` | 构造 `{"role": "assistant", "content": [{"type": "tool_use", "id": call_id, "name": name, "input": parsed_arguments}]}` |
| `make_tool_result_message` | 构造 `{"role": "user", "content": [{"type": "tool_result", "tool_use_id": call_id, "content": output}]}` |
| `extract_tool_calls` | 从 `message["content"]` 列表中提取 `type: "tool_use"` 的 block，拼接 `type: "text"` block 的文本 |
| `detect_stream_tool_boundary` | 返回 `None`（Anthropic 工具调用通过结构化事件传递） |

## 9. 上游客户端

### 9.1 设计

使用单一 `UpstreamClient` 类处理 OpenAI 和 Anthropic 两种上游，通过 `ProviderConfig` 区分鉴权方式和端点路径。

```python
class UpstreamClient:
    def __init__(self, config: ProviderConfig):
        self.config = config
        self._client = httpx.AsyncClient(
            timeout=httpx.Timeout(config.timeout_seconds, connect=10.0),
        )

    async def send(self, body: dict) -> dict:
        """发送非流式请求，返回解析后的 JSON 响应"""

    async def send_stream(self, body: dict) -> AsyncIterator[SseEvent]:
        """发送流式请求，yield 已完成分帧的 SSE 事件

        这是一个 async generator。调用 `client.send_stream(body)` 时会返回
        一个 `AsyncIterator[SseEvent]`，供 converter 直接 `async for`
        消费，不需要额外 `await`。
        """

    async def close(self):
        """关闭底层 HTTP 客户端"""
```

### 9.2 端点路径

根据 `protocol` 和 `wire_api` 确定：

| protocol | wire_api | 端点路径 |
|---|---|---|
| `openai` | `chat` | `{base_url}/chat/completions` |
| `openai` | `responses` | `{base_url}/responses` |
| `anthropic` | `messages` | `{base_url}/v1/messages` |

### 9.3 鉴权头

| protocol | Header |
|---|---|
| `openai` | `Authorization: Bearer {api_key}` |
| `anthropic` | `x-api-key: {api_key}` + `anthropic-version: 2023-06-01` |

`api_key` 解析优先级：`config.api_key` > `os.environ[config.env_key]`。所有鉴权头构造只能调用 `config.resolve_api_key()`，禁止在其他模块重复拼接读取逻辑。

### 9.4 重试策略

仅在 `send` 和 `send_stream`（建立连接阶段）中重试，最多 `max_retries` 次。

可重试的错误：

| 错误类型 | 重试条件 |
|---|---|
| `httpx.ConnectError` | 尚无数据接收时重试 |
| `httpx.ConnectTimeout` | 尚无数据接收时重试 |
| HTTP 429 | 重试 |
| HTTP 500/502/503 | 重试 |

不重试的错误：

- HTTP 400/401/403/404（客户端或配置问题）
- 流式传输中途断开（已有部分数据，无法安全重试）
- `httpx.ReadTimeout`

退避策略：首次重试不等待，后续重试按指数退避（1s、2s、4s、...）。

### 9.5 客户端生命周期

- 应用启动时为每个 provider 创建一个 `UpstreamClient` 实例，存入 `app.state.upstream_clients`
- 应用关闭时调用所有实例的 `close()` 方法
- 同一 provider 的所有请求共享同一个 `httpx.AsyncClient`（利用连接池）

## 10. SSE 工具函数

`sse.py` 提供底层 SSE 操作的工具函数，供 converter 和 upstream client 使用：

```python
@dataclass
class SseEvent:
    event: str | None
    data: str
    raw_lines: list[str]

def format_sse(data: str, event: str | None = None) -> str:
    """格式化单条 SSE 消息

    Args:
        data: data 字段内容（通常是 JSON 字符串）
        event: 可选的 event 类型（如 "response.created"）

    Returns:
        完整的 SSE 文本块，以 \\n\\n 结尾
    """

def format_data_only_sse(data: dict) -> str:
    """格式化仅含 data 行的 SSE 消息（OpenAI Chat 格式）"""

def format_named_sse(event: str, data: dict) -> str:
    """格式化含 event + data 行的 SSE 消息（Responses / Anthropic 格式）"""

def parse_sse_line(line: str) -> tuple[str | None, str | None]:
    """解析单行 SSE 内容

    Returns:
        (field_name, field_value)
        如 "data: {...}" -> ("data", "{...}")
        如 "event: message_start" -> ("event", "message_start")
        空行或注释行 -> (None, None)
    """

async def parse_sse_events(
    lines: AsyncIterator[str],
) -> AsyncIterator[SseEvent]:
    """将原始 SSE 行流组装为事件帧

    规则：
    - 以空行作为一个事件帧结束
    - 同一帧内可包含 0 或 1 条 event 行，以及 1 到多条 data 行
    - 多条 data 行使用 "\\n" 拼接
    - 注释行和空白行不进入 data 内容，但保留在 raw_lines 便于排障
    """
```

### 10.1 帧级抽象约定

所有流式链路都以 `SseEvent` 作为 converter 的输入，而不是原始文本行。

职责边界如下：

- `UpstreamClient.send_stream()`：负责建立上游 HTTP 流连接、读取文本行、调用 `parse_sse_events()` 做分帧，并产出 `SseEvent`
- `Converter.convert_stream()`：负责把上游 `SseEvent` 映射为客户端协议要求的事件序列
- `format_*_sse()`：负责把客户端事件对象编码为最终返回给浏览器或 SDK 的 SSE 文本块

这样每条 converter 只处理“协议事件”，不重复处理空行分隔、跨行 data 拼接等底层细节。

## 11. 错误处理

### 11.1 错误类型层次

```python
class GatewayError(Exception):
    """网关错误基类"""
    status_code: int = 500
    error_type: str = "server_error"
    code: str = "internal_error"
    message: str

class ConfigError(GatewayError):
    """配置类错误：provider 不存在、模型不支持、API key 缺失"""
    status_code = 400
    error_type = "invalid_request_error"
    code = "config_error"

class ProtocolError(GatewayError):
    """协议类错误：请求体结构非法、必填字段缺失"""
    status_code = 400
    error_type = "invalid_request_error"
    code = "invalid_request"

class UpstreamError(GatewayError):
    """上游类错误：连接失败、超时、重试耗尽"""
    status_code = 502
    error_type = "upstream_error"
    code = "upstream_error"

class UpstreamHTTPError(UpstreamError):
    """上游 HTTP 错误，携带原始状态码和响应体"""
    def __init__(self, upstream_status: int, upstream_body: str): ...

class ConversionError(GatewayError):
    """转换类错误：响应解析失败、流中断、工具调用语义非法"""
    status_code = 502
    error_type = "conversion_error"
    code = "conversion_error"
```

常用具体错误码约定：

| 错误场景 | 异常类型 | `code` |
|---|---|---|
| provider 不存在 | `ConfigError` | `unknown_provider` |
| model 不在白名单 | `ConfigError` | `model_not_supported` |
| API key 缺失 | `ConfigError` | `api_key_not_configured` |
| 请求体缺少必填字段 | `ProtocolError` | `missing_required_field` |
| 请求体结构非法 | `ProtocolError` | `invalid_request_body` |
| 上游返回 4xx/5xx | `UpstreamHTTPError` | `upstream_http_error` |
| 上游连接失败或超时 | `UpstreamError` | `upstream_connect_error` |
| 流式中途中断 | `ConversionError` | `stream_interrupted` |
| 工具调用畸形 | `ConversionError` | `malformed_tool_call` |
| 响应解析失败 | `ConversionError` | `invalid_upstream_response` |

### 11.2 错误响应格式

错误响应必须与客户端请求的协议格式一致：

**OpenAI 格式（Chat 和 Responses 通用）：**

```json
{
    "error": {
        "type": "invalid_request_error",
        "code": "unknown_provider",
        "message": "provider 'xxx' not found in configuration"
    }
}
```

**Anthropic 格式：**

```json
{
    "type": "error",
    "error": {
        "type": "invalid_request_error",
        "message": "provider 'xxx' not found in configuration"
    }
}
```

格式化函数根据 `inbound_format` 选择对应格式：

```python
def format_error_response(
    error: GatewayError,
    inbound_format: ProtocolFormat,
) -> JSONResponse:
    if inbound_format == ProtocolFormat.ANTHROPIC_MESSAGES:
        body = {"type": "error", "error": {"type": error.error_type, "message": error.message}}
    else:
        body = {"error": {"type": error.error_type, "code": error.code, "message": error.message}}
    return JSONResponse(status_code=error.status_code, content=body)
```

### 11.3 流式错误处理

流式响应始终以 HTTP 200 开始，错误通过协议对应的完成事件传达：

- **OpenAI Responses**：在 `response.completed` 事件中设置 `status: "failed"` 或 `"incomplete"`，附带 `error` 对象
- **OpenAI Chat**：发送包含 `finish_reason: null` 的最终 chunk，随后 `data: [DONE]`（上游原始错误信息记录日志但不直接暴露给客户端）
- **Anthropic**：发送 `error` 事件，格式为 `{"type": "error", "error": {"type": "...", "message": "..."}}`

### 11.4 统一错误路径

为避免不同 converter 各自定义错误行为，所有请求统一遵循下表：

| 阶段 | 示例 | 非流式处理 | 流式处理 |
|---|---|---|---|
| 配置校验前 | TOML 缺失、provider 未加载 | 应用启动失败，不接受请求 | 应用启动失败，不接受请求 |
| 请求校验 | `model` 缺失、字段结构非法 | 直接返回协议格式错误响应 | 在进入 `StreamingResponse` 前抛出，仍返回普通错误响应 |
| 请求转换 | 字段映射失败、工具参数非法 | 返回协议格式错误响应 | 在进入 `StreamingResponse` 前抛出，仍返回普通错误响应 |
| 上游建连或首包前 | DNS 失败、HTTP 401、429、500 | 返回协议格式错误响应 | converter 发送一条协议内失败事件后正常结束流 |
| 上游流进行中 | 连接中断、读超时、半包 | 不适用 | converter 发送 `incomplete` 或 `error` 终止事件，不再重试 |
| 响应转换 | 上游 JSON 非法、工具调用畸形 | 返回协议格式错误响应 | 若已发送部分内容，则产出协议内 `incomplete` 终止事件；否则发送协议内失败事件 |

补充约束：

- 只有“尚未进入流式响应”时，才允许返回普通 JSON 错误响应和非 200 状态码
- 一旦任意客户端 SSE 事件已经发出，后续错误只能编码到协议自身的错误事件或完成事件中
- 上游原始错误正文只记录到日志；返回给客户端的错误信息必须是网关归一化后的 `type/code/message`
